Round 1

map vs. unordered_map

std::map — Red-Black Tree

  • Guaranteed O(log n) lookup, insert, delete
  • Keys always sorted
  • Node-based: pointer-chasing, cache-unfriendly
  • Stable iterators/references across all operations

Michael Sambol — youtu.be/qvZGUFHWChY

This is what we all learned. It works. It's predictable. But log(n) adds up at scale.

What is a Red-Black Tree?

Self-balancing rules:

✓ Each node: red or black

✓ Root & leaves (nil): black

✓ Red nodes → only black children

✓ All paths: same black node count

Result: O(log n) guaranteed

█
(nil children below each black node)
→ Root (black) + Red child + Black leaf + Nil (black) = 3 black nodes per path ✓
These simple rules prevent the tree from becoming a long chain. As elements are added or removed, the tree rotates to maintain balance. Data storage: Actual data lives in the colored nodes (root, red, black). Nil nodes aren't stored in memory—they're a theoretical concept. In code, we use NULL pointers to represent no child here. The nil nodes help us understand why the balancing rules ensure O(log n) performance.

std::unordered_map — Hash Table

  • Average O(1) lookup, insert, delete
  • Keys unordered
  • Bucket-based: separate chaining (linked lists)
  • "Average" is doing heavy lifting in that sentence
Emphasis on average-case — this is foreshadowing for the pitfalls section.

Under the Hood

How std::unordered_map really works

The Pipeline

Step 1: Key → std::hash → size_t

Step 2: Hash → hash % bucket_count() → Bucket Index

Step 3: Bucket → Linear scan → Data

Three steps. Key goes in, hash comes out, modulo gives us a bucket, we scan the bucket for our key.

Each bucket is the head of a linked list.

When two keys hash to the same bucket, they get chained.

When keys collide, they chain together in a linked list at that bucket. This sets up slide 19's critical insight: performance depends entirely on whether collisions are scattered across buckets or concentrated in one. The rest of the presentation explores what causes buckets to overflow with collisions.

O(1) assumes each bucket has ~1 item.

O(n) happens when one bucket has ALL items.

This is the entire talk in two sentences. Everything that follows is about what makes buckets overflow.

Load Factor


load_factor = size() / bucket_count()

When load_factor > max_load_factor (default: 1.0)

REHASH: Allocate new bucket array → re-hash every element → re-insert all
The standard tries to keep buckets short by rehashing when the table gets too full. But rehashing itself has costs.

Pitfall 1: The Bad Hash

You need to hash a custom struct.

What could go wrong?

The Transaction Struct


struct Transaction {
    std::string merchant_id;
    std::string card_bin;
    uint32_t amount_cents;
    std::string currency;

    bool operator==(const Transaction& other) const {
        return merchant_id == other.merchant_id
            && card_bin == other.card_bin
            && amount_cents == other.amount_cents
            && currency == other.currency;
    }
};

Bad Hash vs Good Hash

BAD 

struct BadHash {
  size_t operator()(
    const Transaction& t) const {
    return std::hash<std::string>{}
      (t.currency);
  }
};
// Only hashes currency
// 99% "USD" → collision!
GOOD 

struct GoodHash {
  size_t operator()(
    const Transaction& t) const {
    size_t seed = 0;
    // Combine ALL fields:
    seed ^= hash(t.merchant_id);
    seed ^= hash(t.card_bin);
    seed ^= hash(t.amount_cents);
    seed ^= hash(t.currency);
    return seed;
  }
};

src/04_pitfall1_bad_hash_txn.cpp

Left: only hashes currency. 99% USD means 99% collision. Right: combines all fields with the golden ratio constant.
No one would hash on a single currency field, but mistakes happen more often then you think.

Things to Think About

Cost

More fields = more computation per lookup. If one field is already unique, hashing the rest is wasted work.

Semantics

Your hash must match operator==. Only hash the fields that define identity.

Mutability

Never hash a field that can change after insertion — find() will look in the wrong bucket.

The Pro-Tip: hash_combine


template <typename T>
void hash_combine(size_t& seed, const T& val) {
    const size_t GOLDEN = 0x9e3779b9;
    seed ^= std::hash<T>{}(val) + GOLDEN
            + (seed << 6) + (seed >> 2);
}

template <typename... Args>
inline size_t hash_values(const Args&... args) {
    size_t seed = 0;
    (hash_combine(seed, args), ...);  // C++17 fold expression
    return seed;
}
    

struct TransactionHash {
    size_t operator()(const Transaction& t) const {
        return hash_values(
            t.merchant_id, t.card_bin, t.amount_cents,
            t.currency, t.timestamp_ms);
    }
};

std::unordered_map<Transaction, std::string, TransactionHash> auth_cache;

Boost-style hash_combine. 0x9e3779b9 is golden ratio. Ensures widely scattered outputs. Knuths. shifts seed left and right, then combines. sSall changes large in final hash. Without XOR its commutative
The variadic fold expression makes it composable for any number of fields.

Beyond unordered_map

C++23 & Modern Alternatives

std::flat_map (C++23)


#include <flat_map>

using BinTable_t = std::flat_map<
    std::string, BinInfo>;
BinTable_t bin_table;
bin_table["411111"] = {"CHASE", "VISA"};
auto it = bin_table.find("411111");
  • Backed by two sorted std::vectors (contiguous memory)
  • Cache-friendly iteration — blazing fast for read-heavy workloads
  • Trade-off: O(n) insert, but BIN tables load once at startup
  • Think: std::map meets std::vector

Heterogeneous Lookup (C++20)


struct StringHash {
    using is_transparent = void;  // Magic marker!

    size_t operator()(std::string_view sv) const {
        return std::hash<std::string_view>{}(sv);
    }
    size_t operator()(const std::string& s) const {
        return std::hash<std::string>{}(s);
    }
};

struct StringEqual {
    using is_transparent = void;

    bool operator()(std::string_view a, std::string_view b) const {
        return a == b;
    }
};

src/10_cpp20_heterogeneous_lookup_txn.cpp

Search for elements in unordered associative containers using different key type.
Both the hasher and equality must declare is_transparent.
StringEqual takes string_view on both sides so it can compare string vs string_view without conversion.
ISO8583 defines credit/debit card transactions. uses bitmap to define fields in use. string_view incredibly efficient.

The Decision Framework

Choosing the Right Container

Three Rules

std::vector is your default sequence.

std::unordered_map is your default associative container.

It is O(1) fast but O(n) dangerous.

1. Bad Hashes → use hash_combine

2. Collisions → hash ALL fields

3. Rehashing → call reserve()

Your New Decision Tree

Need to store items?

├── Sequence? ──────────────────────── std::vector (done)

└── Key-Value?

├── Need sorted order? ──YES──── std::map

├── Need reference stability? ── std::map

└── Otherwise ──────────────────── std::unordered_map

├── Custom key? ──── Write a good hash (hash_combine)

└── Know the size? ── Call reserve()
